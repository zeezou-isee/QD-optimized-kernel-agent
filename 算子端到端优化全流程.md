# **端到端算子宏微观解耦优化全流程**

本节以移动端 ARM CPU 上的 GEMM (General Matrix Multiply) 算子为例，详细阐述其在 Path A（优化路径）中，如何通过“LLM 先验筛选 -> MAP-Elites 宏观探索 -> TPE 微观收敛”的三级火箭架构达到性能极限。

## **阶段一：任务解析与先验空间筛选 (Task Parsing & Prior Pruning)**

**1. 算子画像与瓶颈诊断 (Operator Profiling & Bottleneck Diagnosis)**

* 系统接收到 Python 算子定义 GEMM(M=512, N=512, K=512)、MNN 框架原生 C++ 代码以及目标硬件 ARM Cortex-A。  
* **LLM 诊断**：Accelerator Agent 计算其算术强度 (Arithmetic Intensity)，判定该任务在 ARM 架构上属于**计算瓶颈 (Compute-Bound) 为主，伴随 L1 Cache 访存瓶颈**。这一诊断结果将直接决定后续 MAP-Elites 采用何种动态行为描述符 (Dynamic BD)。

**2. 先验边界裁剪 (Prior Boundary Pruning)**

* 基于 ARM NEON 指令集特性，Accelerator Agent 提出**硬性约束 (Hard Constraints)**：  
  * *约束 1*：禁用一切超过 8 路以上的过度循环展开（防止 Register Spilling）。  
  * *约束 2*：禁用 GPU 特有的 Shared Memory 策略，转而锁定 CPU 特有的 Cache Blocking（L1/L2 缓存分块）策略。  
* **结果**：宏观策略空间被大幅度裁剪，剔除了会导致编译失败或必然降速的死区。

## **阶段二：动态 MAP-Elites 档案库初始化 (Dynamic MAP-Elites Archive Setup)**

基于阶段一的“计算为主”诊断，系统为该 GEMM 算子实例化特定的二维特征网格（Archive），采用**上下文感知的行为描述符（Context-Aware BD）**：

* **BD 1 (缓存阻塞层级)**: [0: 无分块, 1: 仅 L1 Cache 分块, 2: L1 + Register 双层分块]  
* **BD 2 (指令级并行度)**: [0: 纯标量, 1: NEON 向量化, 2: NEON 向量化 + 适度循环展开]

此时，形成了一个 3 * 3 = 9 个生态位（Niches）的网格。在**冷启动阶段 (Cold-Start)**，Accelerator Agent 会采用“意图驱动 (Intent-driven)”策略，主动生成 3-4 份不同风格的初始模板投入网格，作为进化的“火种”。

## **阶段三：宏微观双层进化循环 (Macro-Micro Co-Evolution Loop)**

这是优化的核心引擎，包含外层（Agent 驱动的交叉/变异）和内层（TPE 驱动的参数收敛）的协同。  
```
graph TD  
    A[开始: 获取算子与硬件信息] --> B[LLM诊断瓶颈与先验空间裁剪]  
    B --> C[依据瓶颈实例化动态 MAP-Elites 档案网格]  
    C --> D[冷启动: 意图驱动生成初始精英]  
      
    subgraph "外层: MAP-Elites 宏观策略探索 (Agent 驱动)"  
        D --> E{选择进化操作}  
        E -- 智能交叉 (Crossover) --> F[挑选 2 个互补精英模板]  
        E -- 智能变异 (Mutation) --> G[挑选 1 个精英模板]  
          
        F --> H[Accelerator & Coder Agent: 语义级融合重构]  
        G --> H  
        H --> I[生成带有占位符的 C++ 宏观模板]  
          
        subgraph "内层: TPE 微观参数收敛 (传统算法驱动)"  
            I --> J[提取超参数空间 e.g. BLOCK_SIZE]  
            J --> K{预算耗尽? e.g. 5次}  
            K -- 否 --> L[TPE 采样一组具体数值]  
            L --> M[编译测速与低保真度验证]  
            M --> N[更新 TPE 概率模型]  
            N --> K  
            K -- 是 --> O[输出该模板的 '极限性能' 与 '最佳参数']  
        end  
          
        O --> P[提取新代码的实际 BD 特征]  
        P --> Q{性能是否击败同 BD 网格的老精英?}  
        Q -- 是 --> R[替换入 MAP-Elites 档案库]  
        Q -- 否 --> S[触发局部自愈或丢弃]  
    end  
      
    R --> T{达到最大进化世代? e.g. 30代}  
    S --> T  
    T -- 否 --> E  
    T -- 是 --> U[输出全局性能最高的最终 Kernel]
```

### **详细执行步骤：**

**Step 3.1: 智能交叉与变异 (Agentic Variation & Crossover)**

* **互斥与正交交叉 (Orthogonal Crossover)**：同一 BD 轴上的策略（如纯标量与向量化）绝对互斥。交叉仅在**正交的 BD 轴之间**发生。例如，Agent 提取 (2,0)（极佳访存）和 (0,2)（极致计算）的精英基因，通过 LLM 语义协商，融合成 (2,2) 模板。  
* **变异的三重效应 (Triple-Effect Mutation)**：对单一精英进行语义级重构，引发以下三种状态跃迁：  
  1. **跨网格跃迁 (BD Transition)**：如在 (1,1) 基础上增加循环展开，代码升维并被放入 (1,2) 网格竞争。  
  2. **同网格洗牌 (In-cell Refinement)**：进行指令重排等底层等价变换，BD 坐标不变，但通过更优写法篡位成为当前格子的新王。  
  3. **降级自愈 (Bottleneck Alleviation)**：主动删减过度激进的优化以消除寄存器溢出，代码退化回低级网格（如 (2,1)），但获得该生态位的最高分。

**Step 3.2: 内层参数收敛 (Micro Parameter Exploitation)**

* 模板被下放给 TPE (Tree-structured Parzen Estimator) 算法引擎。  
* TPE 识别出参数空间：BM ∈ [16, 32, 64], UNROLL ∈ [2, 4]。  
* TPE 开始在真机上进行极小预算的快速试错（例如只试 5 次）。  
  * *尝试 1*: BM=64, UNROLL=4 -> OOM，耗时 5.2ms。  
  * *尝试 5*: BM=16, UNROLL=4 -> 最佳状态，耗时 **1.8ms**。  
* **交付结果**：内层确认该宏观模板的“局部性能天花板”为 **1.8ms**。

**Step 3.3: 质量-多样性归档 (Quality-Diversity Mapping)**

* 系统拿着这个 (1.8ms, 配置信息) 的包裹，使用静态分析或 LLM 提取其实际的 BD 特征。  
* 假设变异后的代码落入了坐标 (2, 2) 的格子。  
* 检查格子：如果格子是空的，或者格子里的老代码耗时大于 1.8ms，则新代码正式成为该生态位的**新精英 (Elite)**。

**Step 3.4: 失败驱动的规则沉淀 (Failure-Driven Constraint Mining)**

* 如果新模板引发了不可恢复的编译失败，Debugger Agent 介入分析报错，提取出一条**不可违背的全局规则 (Global Negative Constraint)**，并动态封锁引发该错误的 MAP-Elites 危险区域。

## **阶段四：收敛与输出 (Convergence & Output)**

* 经过 30 代进化，MAP-Elites 网格被各种流派的高分代码填满。  
* 系统挑选出**全局延迟最低 (Global Best)** 的组合，具体化为最终 C++ 源文件。  
* **跨任务收益**：整个档案库作为“技能兵器谱”被保存，供未来优化类似算子时进行 RAG 检索与热启动。
