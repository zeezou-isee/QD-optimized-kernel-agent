# 算子优化:从问题来源到 LLM 驱动的自动化体系

> 本文档总结一次完整的设计讨论:从"算子优化是什么"出发,逐步把它形式化为一个**结构化空间上的昂贵黑盒优化问题**,
> 厘清其难点与可利用结构,最终落到一套**以 LLM 做 kernel generation** 为核心、由 **Proposer / Evaluator / Policy** 三角色
> 配合的自动化体系。
>
> 应用背景:面向移动设备的推理框架(类 MNN / NCNN),需要 **Vulkan 后端 + CPU(ARM)后端**。

---

## 0. 一句话主线

> 针对**特定算子 + 特定硬件**,把"该施加哪些优化、各用什么参数"建模为一个**离散、条件结构、强约束**空间上、
> 以**真机运行时**为目标的**昂贵黑盒优化**;用 **LLM 当 Proposer(结构层) + 真机实测当 Evaluator(参数层) + 两层异构搜索当 Policy**,
> 并用 **roofline 定向/定停** 减少无效探索;参数层提供**参数化 kernel 扫描(主选)/ LLM 迭代生成配置(备选)**两种遍历方案。
>
> (注:本轮起将 LLM 调用与实测均视为**零成本**,核心问题聚焦"**减少无效组合**"。)

---

## 1. 问题来源:算子优化有哪些方法

算子优化是深度学习编译器 / 高性能计算的核心。常见方法可按**优化目标**归类:

- **降计算量 / 提计算效率**
  - 代数化简、强度削减、表达式重排
  - 算法替换:Winograd、FFT 卷积、Strassen、低秩分解
  - 算子融合(conv+bn+relu、残差融合):减少 kernel launch 与中间结果落地
  - 量化 / 低精度(INT8 / FP16 / BF16)
  - 稀疏化
- **降访存(移动端的主战场)**
  - Tiling / 分块(适配 cache / shared memory / 寄存器)
  - 数据布局变换(NCHW↔NHWC、im2col、packing、NC4HW4 等 channel 分块布局)
  - 局部性优化(loop interchange)、寄存器/shared memory 复用
  - 访存合并(coalescing)
- **提并行 / 隐藏延迟**
  - 循环变换(unroll、fuse/fission、reorder)
  - 向量化(SIMD:NEON / AVX)
  - 多级并行映射(grid/block/thread/warp、tensorize 到 TensorCore/MMA)
  - 流水线 / 双缓冲(double buffering),计算与访存 overlap
  - 负载均衡(split-K)
- **硬件指令 / 微架构层**
  - 专用指令(TensorCore、dp4a、dotprod)、bank conflict 消除、ILP、寄存器分配

### 分类的多个正交维度
| 维度 | 类别 |
|---|---|
| 优化目标 | 降计算 / 降访存 / 提并行 / 隐藏延迟 |
| 抽象层次 | 图级(graph)/ 循环级(loop)/ 指令级(instruction) |
| 瓶颈类型(roofline) | compute-bound / memory-bound |
| 是否改变数值结果 | 严格等价 / 近似(量化、近似算法) |
| 谁来做 | 手工 / 编译器自动 |

业界主线是 **图级 → 循环级 → 指令级** 的分层 lowering(TVM / MLIR / XLA)。

### 能否穷举?
- **优化"手段"**:在**固定硬件 + 框架**下基本可枚举(有限的调度原语集)。
- **优化"方案空间"**(手段组合 + 参数):**组合爆炸**,不可暴力穷举,只能搜索。
- **优化"方法"作为学科**:随硬件演进**开放增长**,不可穷举。

## 2. 落到具体场景:移动端推理框架(CPU + Vulkan)

移动端的约束主导一切:**ARM CPU + 移动 GPU(Adreno/Mali/PowerVR)**、功耗发热、内存带宽小、统一内存(UMA)、cache 很小。

### 优化矩阵(近似分类)
| | CPU(ARM) | Vulkan(移动 GPU) |
|---|---|---|
| 降计算 | Winograd、量化、NEON dot(sdot/udot) | fp16、算法选择 |
| 降访存 | packing、tiling、fp16 | image 布局、local memory、vec4、融合 |
| 提并行 | 多线程 + 绑核、SIMD | workgroup 调优、合并访存 |
| 隐藏延迟 | 流水线汇编 micro-kernel | command buffer 批量提交、减少同步 |

### 路线选择
- **TVM/Ansor 路线**:原语 + 大搜索 + 自动调优 → 适合通用编译器。
- **MNN/NCNN 路线**:专家手写有限 kernel 变体 + 启发式派发 → 小体积、可预测、易维护。
- **移动端关键**:autotuning 是**开发期工具**,产出的是**出厂即用的固定 kernel**(不在用户手机上搜)。

## 3. 把问题形式化为组合优化

### 3.1 候选方案的表示
一个候选 = 一组决策:
```
c = (s1, p1, s2, p2, ...)
  s_i:离散/类别选择(用不用 Winograd、buffer 还是 image、哪种 tiling)
  p_i:该选择下的参数(tile 大小、unroll 因子、workgroup size、线程数)
```

### 3.2 优化空间的构建:一棵树
有两种等价视角:
- **配置树**:节点 = 部分决策,叶子 = 完整配置(只有叶子可跑)。
- **施加树(主用)**:节点 = **一个已能跑的 kernel**,深度 = **已叠加的优化数量**;**每个节点都可实测**。

> 两者统一:施加树中"深度 d 的节点" ≡ 配置树中"d 个旋钮非 off、其余全 off 的叶子"。
> 由此,"组合长度可变" 这个表面难点被**规范化为定长 + 空值(off)**,不是本质难点。

**这棵树的形态**:
- **算法层(L1)**:分支少(direct / im2col / Winograd…,约 3–5);**兄弟差距大**(不同 basin)。
- **参数层(深层)**:分支多(tile/unroll 取值);**兄弟差距小**(局部光滑),偶有悬崖。
- **子树深度不齐**:Winograd 子树更深(多变换决策),elementwise 子树很浅。

**真正的结构性难点(消解不掉的)**:
1. **条件依赖**:选了某分支才解锁某些参数维 → 空间是树状/条件,不是规整笛卡尔积。
2. **顺序敏感**:少数循环变换的结果依赖施加顺序。

### 3.3 优化问题的定义
```
min_{c ∈ F}  Latency(c)        F = 可行域(满足约束的配置)
```
评判真值 = **真机实测运行时**(亦可为延迟/能耗)。

### 3.4 难点:为什么不是教科书式组合优化
目标函数有四个性质,决定了不能直接套经典求解器:
1. **混合空间**:类别 + 整数参数 + 有时顺序。
2. **条件/层次结构**(树状)。
3. **强约束、大量非法点**:tile 须整除、须放进 shared memory、寄存器不能溢出。
4. **目标是昂贵、带噪、无解析式的黑盒**:每次求值 = 编译 + 真机 + 计时,且受温度/DVFS 噪声影响。

> 第 4 点是与经典组合优化(TSP/背包/ILP,目标可解析)的本质区别。
> 准确命名:**结构化空间(混合/条件/约束)上、昂贵黑盒目标的优化**。

## 4. 关键认知:landscape 是结构化的,不必穷举

对"只能遍历整个空间吗?"的回答是**否**。性能 landscape **不是随机的**,而是
**带噪 + 分段光滑 + 有悬崖 + 非凸**,因为它被**硬件物理**支配。

### 4.1 分段光滑 + 悬崖的来源(主要在参数轴)
- **分段光滑**:同一"机制状态"(工作集同在 L1、同一 occupancy 档、同一对齐类)内,参数变一点,性能缓慢变 → **这就是"前进方向"**。
- **悬崖(突变)**:离散硬件资源越过硬上限时整体跳档:
  - cache / shared memory 容量溢出 → 落更慢的存储层
  - 寄存器溢出(spill)→ local memory
  - occupancy 量子化(warp/block 按离散单位分配,跨档则并发整数级下降)
  - 对齐:是向量宽度/cache line 整数倍则全向量化,差 1 则需标量尾巴

### 4.2 深度 → 收益:非单调
- **边际递减**:头几个优化吃掉大部分收益。
- **负交互(关键)**:优化间会打架(如激进 unroll 抬高寄存器压力 → 触发 spill → 反而更慢)。
  "加优化 X 的收益"依赖"已加了什么"(**非可分 / non-separable**)。
- **推论:最优常在中间深度**——"堆满所有优化" ≠ 最优,最优在"边际收益转负"处。

### 4.3 兄弟差距
- 参数分叉的兄弟:**差距小**(光滑),例外是跨悬崖。
- 结构/算法分叉的兄弟:**差距大**(不同 basin)。

### 4.4 为什么"可搜索"有理论依据
**No Free Lunch**:仅当目标函数**任意随机**时,所有优化器才等价(只能穷举)。
本问题目标受 roofline / cache / occupancy 约束,**高度结构化**,故 NFL 不适用 → **引导式搜索能显著优于穷举/随机**。
关键性质:**性能"可近似、可排序",而非"可精确预测"**——搜索只需"A 大概率比 B 快",这弱要求可满足。

## 5. 建模为三角色体系

把系统拆成三个**正交、可替换**的角色:

| 角色 | 职责 | 谁来当 | 可靠性 |
|---|---|---|---|
| **Proposer 提议者** | 给定节点,产出合法子节点(下一步可施加的优化) | LLM(算法层) + 多面体/原语集(调度层) | 广,但需验证 |
| **Evaluator 评判者** | 一个节点有多好 | **真机实测**(真值) + 可选学习型代价模型 | 实测可靠 |
| **Policy 策略** | 展开/探索哪个节点、何时停 | 搜索算法,受 roofline 先验引导 | —— |

### 5.1 roofline 的三重作用
roofline 不只给瓶颈类型,而是给:
1. **天花板**(算力/带宽上限)
2. **当前位置**(实测点)
3. **离天花板的余量 + 撞哪面墙** ← 最有用,用于**剪枝定向**

注意:**瓶颈类型依赖实现**(naive conv 是 memory-bound,tiling 充分后可能变 compute-bound)。
- memory-bound → 减 FLOP 的优化(Winograd)基本无效,走访存分支(layout/fusion/tiling/fp16)。
- compute-bound → 走算力分支(向量化、TensorCore/dotprod、Winograd)。

### 5.2 树的方向是否依赖人的知识?(分层回答)
- **调度层(tiling/unroll/vectorize/reorder/fusion/layout)**:**可机械化、可系统枚举**。
  多面体模型(polyhedral)用数学**推导所有合法仿射变换**,不靠人列清单;或固定原语集亦有限可枚举。
  → 这一层基本**不会漏分支**。
- **算法层(Winograd/FFT/im2col/低秩)**:**数学重表述,不可避免依赖知识**。
  但好在是一个**小而明确的已知目录**;风险是"漏掉一个**已知**技术",可用好的知识库 / LLM 兜底。
- 结论:树的方向**部分依赖知识,且分层、有界**,并非"全靠人、随时可能漏"。

## 6. LLM 做 kernel generation:它在体系中的位置

### 6.1 LLM 的角色边界
- **LLM = Proposer**:广博但不可靠的知识库,擅长列算法/调度选项、写 kernel 代码。
- **LLM ≠ Evaluator**:**不能可靠预测运行时**,绝不让它拍脑袋判快慢 → 真值交给真机。
- **必须配验证器**:LLM 会幻觉出不合法/不存在的变换 → 每个提议过"语义保持?能否编译?硬件合法?"。
- **roofline 注入提议环节**:先告诉 LLM 瓶颈与余量,让它**提议时就剪枝**(别无差别列举)。
- **懒展开(lazy expansion)**:树是条件结构,只展开决定要探的节点的孩子(MCTS / best-first 思路),不全量物化。

### 6.2 成本假设修订(本轮):LLM 与实测均视为零成本
本轮起,**把 LLM 调用与真机实测都当成零成本**,以剥离"贵/便宜"的不对称,聚焦真正的核心问题:

> **中心问题:如何减少"提议的优化方向 / 组合"中的无效组合(无效 = 不会带来增益的方向)。**

即使单次探测零成本,空间巨大、探测次数有限,**降低无效提议占比 = 提升在有限搜索内命中真优解的概率**。
因此重心从"省成本"转为:**Proposer 提议得准(少无效方向)+ Policy 不在死路上浪费迭代**。

说明:下文 6.3–6.5 中以"成本不对称"为动机的论证,其**结论仍有效**(参数化 kernel、roofline 过滤),
但**动机改为"结构化搜索 + 减少无效组合"**,而非省钱。

### 6.3 代价模型补充说明
- **解析型(白盒)**:基于硬件公式(roofline、cache 模型、occupancy)。零数据、即时、可解释;难抓脏细节与悬崖。
  (MNN/NCNN 的启发式派发本质就是轻量解析代价模型。)
- **学习型(黑盒)**:数据训回归(XGBoost / NN / **ranking 模型**)。能抓脏细节;但需大量实测数据、跨硬件难迁移。
- **对本场景的取舍**:**基本不用学习型**;只用**轻量解析模型(roofline)当"提议过滤器"**,决定哪个结构分支值得花一次 LLM。

### 6.4 参数空间的两种遍历方案(主选 + 备选)
> 二者皆保留;主选实现更省迭代,备选实现更简单,对"不确定怎么搭扫描 pipeline"时更友好。
> 两方案对外**输出相同(basin 价值)**,Policy 外层无需区分。

**方案 A(主选):参数化 kernel + 扫描**
LLM **一次生成一个参数化 kernel**(tile/unroll/workgroup 做成可配置常量 / **Vulkan specialization constant**),整个参数子空间用扫描去探:
- **参数化负担甩给 LLM**:写进 codegen 契约(每个 kernel 把旋钮做成 spec constant),不用手工改。
- **Vulkan 上近乎免费**:spec constant 是为此设计的特性,创建 pipeline 时塞常量,驱动重新优化 SPIR-V(常量折叠/死代码消除),无需重生成源码。
- **CPU 上**:模板/宏 + 重编译,标准 C++ 模板。
- **选参数不难**:LLM 顺手给候选值 + 局部光滑 → 粗网格 + 局部微调 + 解析剪枝去非法,**无需专门搜索算法**。

**方案 B(备选):LLM 迭代生成具体配置(LLM-as-optimizer)**
内层不做参数化,而是**让 LLM 直接生成它认为最好的配置,每轮一个,循环若干次**。
- **关键增强**:每轮把"**已试过的配置 + 各自实测延迟**"回喂给 LLM,让它做**有反馈的优化**(类 OPRO:以历史轨迹为条件提议更优点),而非盲目重生成 → 越生成越准。
- 优点:实现简单,无需搭参数扫描 pipeline;缺点:依赖 LLM 的参数直觉,收敛性不如系统扫描确定。

**减少无效结构组合的过滤器(A/B 通用)**:
- 先用 **roofline** 砍掉不该走的分支(memory-bound 不生成 Winograd)。
- 让 LLM 提议时**附带"为什么该有增益"的理由**,据此过滤明显无效 / 互斥(负交互)的组合。

### 6.5 摊薄论证(决定值不值得建基建)
> (注:本轮已将成本视为零,本节"省成本"动机退居次要;但"测量 harness 是地板""基建可摊薄"作为**工程事实仍成立**,
> 现在选 A/B 方案更多看**实现复杂度**而非成本。)
- 测量 harness 是**任何方案都躲不掉的地板**(纯对话式 LLM-in-loop 也要"编译+跑+计时+反馈")。
- 参数化方案的**额外**部分只有:让 kernel 带旋钮(甩给 LLM)+ 一个 spec-constant 扫描驱动(小)+ 解析剪枝(小)。
- **做框架 = 几十算子 × 多设备 × 多参数**,基建一次投入、海量复用 → **强烈值得建**。
- 若只优化一两个 kernel(验证可行性),则轻量对话式循环更划算,验证通了再上基建。

## 7. Policy 设计:怎么"走"这棵树

### 7.1 核心原则:两层异构搜索
因为结构层与参数层在**成本**(LLM vs 实测)和**几何**(跳变多 basin vs 光滑有悬崖)上都不同,
**不能用一套统一搜索走全树**。

### 7.2 内层:参数层(实测驱动)
**方案 A(主选,可近乎穷举)**:
```
1. LLM 生成 kernel 时顺带吐候选参数值
2. 解析剪枝:去非法点(超 shared mem / 寄存器溢出 / 不整除 / 不对齐)→ 不进真机
3. 粗网格实测
4. 在最优点附近局部爬山 / 坐标下降(利用局部光滑)
→ 输出:该 basin 的最优延迟 = 此结构分支的"价值"
```
**方案 B(备选,LLM-as-optimizer)**:
```
循环若干轮:LLM 基于"历史(配置, 实测延迟)"提议一个它认为最好的配置 → 实测 → 回喂
→ 取历史最优作为该 basin 的"价值"
```
两方案输出相同(basin 价值),外层 Policy 无需区分用了哪个。
副作用:实测零成本 → 内层可近穷举,basin 价值低噪声 → **大幅简化外层**(经典 bandit 的估计噪声问题基本消失)。

### 7.3 外层:结构层(贵,LLM 驱动,roofline 引导的 beam search)
```
① Seed:roofline 定向 → 一次 LLM 批量产出 N 个多样化结构候选
② 对每个候选跑内层 → 得 basin 价值
③ Beam:保留 top-k
④ 扩展:让 LLM 针对 top-k 提议"变体 + 协同组合",并反馈"已试过什么 + 实测延迟 + 离 roofline 差距"
⑤ 回到 ②,直到停止
```
三个设计要点:
1. **不用纯贪心叠加,要让 LLM 提"协同组合跳"**:负交互会让纯贪心困在局部最优;
   LLM 知道哪些优化协同 → **用领域知识替代"靠实测发现优化间交互"的昂贵探索**(LLM 在此角色的最大价值)。
2. **深度搜索不用"一退即停"(本轮修正)**:单步变差**不立即剪枝**,用"耐心 + 最优回溯 / best-first"避免被负交互和局部最优困住。详见 7.4。
3. **roofline 当全局停止信号(本领域奢侈品)**:逼近天花板(差距 < ε)就停,已榨干可达性能。

### 7.4 停止条件与深度搜索(本轮修正)
**错误做法**:"子节点不如父节点就立即停 / 剪枝"。它太贪心,会被三件事坑:
(1) 局部最优;(2) **负交互**——有益组合可能要先经过一个暂时变差的中间步;(3) 我们已知**最优常在中间深度**,但通往它的路径未必单调。

**修正做法(耐心 + 最优回溯,推荐用 best-first 实现)**:
- **warmup**:前几层几乎必涨,不启用停机逻辑。
- **耐心 p**:允许连续 p 步深化未刷新全局最优,才放弃该路径(即 ML 早停的 patience)。
- **保留全局最优 + 回溯**:放弃路径时回到全局最优,从那里(或其他前沿节点)换方向继续。
- **边际窗口阈值**:全局最优在最近 W 步的提升 < ε 时,判定边际递减 → 全局停。
- **best-first 前沿(推荐实现,泛化了回溯)**:维护一个**有界优先队列(beam 宽 W)**,按实测价值排序,
  总是展开当前最有希望的未展开节点的子节点。深路径不再增益时,较浅的好节点自然升到队首被展开 →
  **天然处理"最优在中间深度",且回溯是其自然行为**。

**与 LLM 组合跳互补**:组合跳直接跨过负交互的"山谷",耐心 / best-first 处理剩余的非单调性。

**全局停止条件**:
- **逼近 roofline** → 已达天花板,停
- **全局最优 K 轮(或边际窗口 < ε)无改进** → 收敛停

### 7.5 为什么贴合本场景约束
- LLM 提议集中在结构节点,**批量 + 带反馈**,信息量最大化(即便零成本,也减少无效轮次)。
- 内层可近穷举,basin 价值低噪声 → 外层探索/利用难题简化。
- 真正难点收敛为:① 结构层**覆盖度/多样性 + 减少无效组合**(别漏好 basin、别空转死路);② **何时停**(roofline + 耐心 / best-first)。

### 7.6 框架级摊薄:跨算子/设备迁移
- 设备 A 的获胜结构+参数 → 设备 B 的**强先验/种子**(热启动,而非从头提议)。
- 相似算子互为先验。
- 在"几十算子 × 多设备"规模上再砍 LLM 与实测次数。

### 7.7 整体形状
```
roofline 定向/剪枝
      │
      ▼
[外层 beam:结构层] ──LLM 批量提议(多样 + 协同组合)──┐
      │ 对每个结构                                    │ 反馈:延迟 + roofline 差距
      ▼                                               │
[内层:参数层] LLM 候选 → 解析剪枝 → 真机粗扫 → 局部爬山 ─┘
      │ basin 价值(低噪声)
      ▼
保留 top-k → 扩展变体 → ... → 停(逼近 roofline / 全局最优 K 轮无改进;
                                  深度用"耐心+best-first 回溯",不一退即停)
      │
      ▼
跨算子·跨设备 缓存复用 + 热启动
```
> 注:7.8 把"外层结构搜索"的骨架从 beam/MCTS **升级为 Quality-Diversity**(理由见下)。
> 7.7 的整体形状仍成立,只是"外层 beam"替换为"外层 QD 档案"。

### 7.8 结构层搜索算法:候选与选择(落点 = Quality-Diversity)

**先排除已饱和的**:
- **MCTS / UCT + 渐进加宽**:契合度最高(懒展开、探索利用、加宽纳新),但**已被他人用于 LLM-kernel 搜索**,新颖性弱。
- **进化 + LLM 变异/交叉**:已被 FunSearch / AlphaEvolve / EoH / Ansor 做透。
- **beam search**:Halide 经典调度骨架,适合当**基线**,不当卖点。

**候选算法速览**:

| 算法 | 角色 | 契合点 | 饱和度 |
|---|---|---|---|
| MCTS/UCT + 加宽 | 树搜索骨架 | 探索利用、懒展开 | 高(已用) |
| A* / 分支限界 + roofline 启发 | best-first / 剪枝 | roofline 当(近似)可采纳下界 | 低(差异化空间) |
| diverse beam | 有界 best-first | 简单可预测 + 相似度惩罚保覆盖 | 中(基线) |
| 进化 / 交叉 + LLM | 组合发现 | 重组找协同、种群多样 | 高(已用) |
| **Quality-Diversity / MAP-Elites** | **照亮档案** | **构造性保多样 + 零成本实测付得起** | **低(更新鲜)** |

**两个独特资产决定落点**:
- **实测零成本** → QD"吃评估"的死缺点消失(别人评估受限,你不受限)。
- **roofline 物理下界** → 既当 BD 一轴(算术强度),又当 B&B 剪枝界。

**推荐骨架(算法细节见附录 A)**:
> **Quality-Diversity / MAP-Elites** 作结构层骨架;**LLM 当变异/重组算子**;**内层参数搜索当 fitness(实测)**;
> **roofline 当 BD 一轴 + B&B 剪枝下界 + 停机**;**经验池播种初始档案(地板,不过滤)**;
> 先验随深度衰减 + novelty 注入;可选进化交叉增强组合多样性。

### 7.9 行为描述符(BD)设计与待筛组合

**选轴三准则**:① 可廉价获得(优先"生成时即知/静态可算",能在实测前分格);② 与 fitness 正交(别让轴变相等于延迟);③ 轴间不冗余(occupancy 与"内存占用"二选一;算法族与 AI 相关但非共线,可共存)。

**候选轴**:

| 轴 | 类型 | 来源/成本 | 捕捉的多样性 | 备注 |
|---|---|---|---|---|
| 算法族 {direct/im2col-gemm/winograd23/43/dw/1×1} | 类别 | 结构,免费 | 根本计算策略 | **最重要**,防大策略塌缩 |
| 算术强度 AI=FLOP/bytes | 连续 | 静态可算 | roofline 位置/瓶颈区 | 按 ridge 归一 |
| 共享内存/寄存器占用 | 连续 | 静态(tile/layout) | occupancy 区间、cache 适配 | GPU 关键 |
| 数据布局 {buffer/image, NCHW/NHWC/NC4HW4} | 类别 | 结构,免费 | 访存模式族 | **Vulkan 关键** |
| 向量化/打包宽度 {1,2,4,vec4} | 离散 | 结构,免费 | SIMD 使用 | CPU(NEON)关键 |
| 并行粒度 {thread/输出, tile/wg, split-K 度} | 类别/离散 | 结构,免费 | 并行映射 | 注意与占用冗余 |
| 精度 {fp32/fp16/int8} | 类别 | 结构,免费 | 性能/精度大区 | 改数值结果,通常当约束 |
| occupancy(实测) | 连续 | 实测,便宜 | GPU 利用率 | 与"占用"冗余,二选一 |

**待筛组合(配实验)**:
1. **①(推荐基线,2D):算法族 × 算术强度** —— 最根本、最正交、全免费/静态。验证"算法族+roofline 位置"是否已够。
2. **②(Vulkan,3D):算法族 × 共享内存占用 × 布局{buffer,image}** —— 瞄准移动 GPU 主导杠杆。
3. **③(CPU/ARM,3D):算法族 × 向量化打包 × tile 工作集相对 cache 档** —— SIMD + cache 分块 + packing。
4. **④(纯物理,跨后端,3D):算术强度 × 内存占用 × 并行粒度** —— 不放显式算法族,验证"物理描述符能否隐式捕捉算法差异"(回答"算法族轴是否必要")。
5. **⑤(富集,CVT,5D):算法族 × AI × 共享内存 × 布局 × 向量宽度** —— ①②③ 太粗时再上。

**分箱/归一化**:AI 按 **roofline ridge point** 归一分箱(左=访存受限/右=计算受限,跨设备可比);占用/内存用 log 尺度;起步粗(每轴 4–6 bin);类别轴天然分格。

**判优指标(筛选这几组)**:① Coverage(填满格子比例);② QD-score(精英 fitness 之和);③ Best-fitness(**关键:多样性是否帮你找到比单峰搜索更好的最优**);④ 最优解来自哪个格子(若来自非主流格子=多样性付费成功)。

**起步建议**:Vulkan 先上组合 ②、CPU 先上组合 ③,均以组合 ① 为对照;单独验证组合 ④ 以回答"算法族轴是否必要"。

## 8. 经验池(Experience Pool):Proposer 的知识底座

经验池的作用:把"优化知识"从 **LLM 的隐性记忆**外化为**显式、可审计、可扩展、可积累**的知识库,
让 LLM 从"知识来源"降级为"在知识库上做检索/组合/适配的推理引擎"。
它同时治两件事:**减少无效组合**(提议得准)+ **不漏有效组合**(覆盖度兜底)。

### 8.1 总原则:需求驱动 + 自下而上 + 分层
1. **schema 由消费方定,不由资料定**:按 Proposer/Policy 实际要消费的东西设计记录格式,再去填。
   消费需求(已知)= ① 结构候选菜单(算子×瓶颈)② 每个候选何时占优(派发条件)③ 硬件特征 ④ 参数种子值 + 约束 ⑤ 历史结果。
2. **能查则查、能测则测、查不到测不出才读文档**:个人研究者**不需要 NDA spec**——关键量要么 Vulkan 运行时能查,要么微基准能测(实测零成本是最大杠杆)。
3. **分层,把"厂商各不相同"的影响关进小盒子**:
   - **设备无关层(大头、稳定)**:算法/调度知识,所有硬件成立,可移植。
   - **设备相关层(很小)**:厂商/架构特性,大多可查/可测。

### 8.2 硬件知识:别追 spec,用"查询 + 微基准"
- **(A) Vulkan 运行时查询(免费、厂商无关)**:`maxComputeSharedMemorySize`、`maxComputeWorkGroupSize/Invocations`、
  `subgroupSize`(VkPhysicalDeviceSubgroupProperties)、`VK_KHR_cooperative_matrix`(TensorCore 等价物)、
  `shaderFloat16` / `VK_KHR_16bit_storage`、各种 alignment/limits。**架构上最影响决策的量,一行 API 拿到。**
- **(B) 微基准探针(自动发现"有效行为"与悬崖)**:stride benchmark 测有效 cache 容量;occupancy 扫描测寄存器压力悬崖;
  fp16/fp32 同 kernel 测吞吐比。**spec 给不了的行为,全部测得出。**
- 结果:每台设备一份 `{查询 limits} + {微基准行为画像}`。**个人可建、厂商无关、不需内部文档。**

### 8.3 Vulkan 7000 页 PDF:不要做全量 RAG
Vulkan spec 是**正确性/API 参考**,不是性能知识源,99% 无关。只需窄窄几块:
compute dispatch、**specialization constant**、descriptor/push constant、设备查询 API、subgroup、cooperative matrix、16-bit 扩展、内存/barrier 基础。
做法:**整理一份几页的"性能相关能力清单",其余靠 LLM 已有 Vulkan 知识 + 运行时查询;PDF 仅当按需参考,绝不预先入库。**

### 8.4 从 MNN/NCNN 后端提取:不是 chunk 代码,是"蒸馏成结构化记录"
最高价值来源(开源、专家手写、正好是目标后端)。**切分单位 = 语义优化单元 =(算子, 后端, 变体)**,每个蒸馏成一条记录:

| 字段 | 提取什么 | 价值 |
|---|---|---|
| 变体菜单 | 实现了哪些策略(NCNN conv:winograd23/43、im2col-sgemm、direct、1×1 专用、depthwise…) | 直接填充"结构候选菜单" |
| **派发启发式** | 按 shape/channel/对齐选变体的 if-else | **皇冠明珠**:专家蒸馏的"何时哪个变体赢" = 现成剪枝/代价规则 |
| layout 约定 | NC4HW4 packing、image vs buffer | layout 候选 |
| 关键参数 | 代码写死的 tile/workgroup 值 | 参数内层搜索的种子值 |
| 代码片段 | shader/glsl、汇编 micro-kernel | 给 LLM 做改写的 few-shot 参考 |

**准则:不切原始代码,而是把代码蒸馏进此 schema(蒸馏过程可让 LLM 读实现、按模板总结)。优先抓"派发启发式"。**

### 8.5 多框架交叉:一致/分歧本身就是知识
同后端→做法相近。跨框架对比同一(算子, 后端)给两个免费信号:
- **多框架一致** → 强"已知良好"信号,进设备无关层兜底。
- **框架间分歧** → 标记"有真实选择/设备相关",**正是搜索该重点探的入口**。

### 8.6 最终形态与建设优先级
```
经验池
├─ 设备无关层(大头,稳定)
│   ├─ 算法知识:算子×瓶颈 → 候选菜单 + 适用条件(蒸馏自 MNN/NCNN/多框架 + LLM)
│   └─ 调度知识:原语/约束(可机械化,部分多面体)
├─ 设备相关层(小)
│   ├─ 查询画像:Vulkan limits(运行时拿)
│   └─ 行为画像:微基准结果(自动测)
└─ 历史层:跨算子/设备的获胜配置缓存(越用越富)
```
**性价比排序(先做哪个)**:
1. Vulkan 运行时查询 + LLM 内置知识(零成本、厂商无关)→ 立刻可用
2. 微基准探针(写一次、每设备自动跑)→ **直接解掉"拿不到 spec"**,且免费
3. 蒸馏 MNN/NCNN 的变体菜单 + 派发启发式(一次性、可 LLM 辅助)→ 价值极高
4. 多框架交叉(在 3 上增量)→ 拿一致/分歧信号
5. Vulkan PDF 精选清单(几页)→ **别碰全量**

> 一句话:经验池 = **查一点(Vulkan API)+ 测一点(微基准)+ 蒸馏一点(开源算子库)+ 攒一点(历史结果)**,
> 由消费需求定 schema、分设备无关/相关两层。两个最重的活(啃 7000 页 PDF、拿 NDA spec)恰好**都不需要做**。

## 9. 结论速览(可作为决策清单)

1. 算子优化方法**可分类**(目标/层次/瓶颈/是否等价/谁来做),手段在固定硬件下**可枚举**,但**方案空间组合爆炸**、学科**开放增长**。
2. 在"固定算子 + 固定硬件"下,可建模为**结构化空间上的昂贵黑盒优化**;空间是**条件结构的树**,变长可规范化为定长 + off。
3. 目标 = **min 真机运行时**;难点 = 混合 + 条件 + 强约束 + **昂贵带噪黑盒**。
4. landscape **结构化(分段光滑 + 悬崖 + 非凸,最优常在中间深度)**,NFL 不适用 → **不必穷举,可引导搜索**;性能**可排序而非可精确预测**。
5. 体系 = **Proposer(LLM 结构层 + 多面体调度层)/ Evaluator(真机实测)/ Policy(两层异构搜索)**,**roofline 全程定向 + 定停**。
6. **本轮成本假设**:LLM 与实测均视为**零成本** → 核心问题转为**减少无效优化组合**;参数层用**参数化扫描(主)/ LLM 迭代生成(备)**两套方案;**roofline + LLM 理由**当无效组合过滤器;深度搜索用**耐心 + best-first 回溯**(不一退即停)。
7. **做框架 → 基建强烈值得建**;基建一次投入、跨算子/设备摊薄,并用**缓存 + 热启动**复用结果。
8. **经验池 = Proposer 的知识底座**:需求驱动定 schema、自下而上填充(查 Vulkan API + 微基准测 + 蒸馏开源算子库 + 攒历史)、分设备无关/相关两层;**不啃 7000 页 PDF、不拿 NDA spec**。
9. **Policy 结构层骨架 = Quality-Diversity / MAP-Elites**(非 MCTS):BD 切生态位、局部竞争**构造性保多样**(根除过拟合)、LLM 当变异算子、roofline 当 BD 轴+剪枝、经验池播种档案(地板不过滤);**零成本实测正好付得起 QD 的评估账**。详见附录 A。

## 附录 A:Quality-Diversity 与 MAP-Elites 详解

### A.1 设计思路:从"优化"到"照亮(illumination)"
- 传统优化目标 `argmin latency`:搜索**收敛到一个峰**,过程中丢弃所有"不同但有潜力"的解 → 多样性塌缩 / 过拟合。
- QD 目标:**填满一个"行为空间"**,每个区域保留"落在该区域的最好解"。**不收敛到一点,而是铺开**。
- 一个反直觉但重要的结论(**stepping stones**):QD 常常**比直接优化更能找到全局最优**——多样的中间解充当"踏脚石",把搜索引向直接目标驱动搜索因"欺骗性(deception)"而到不了的高性能区。所以多样性既是目的,也帮你找到那个单一最优。

### A.2 两个核心要素(必须分清)
1. **质量 / fitness**:真正在乎的目标 = **实测延迟**(越低越好)。
2. **行为描述符 / BD**:刻画一个解"长什么样/怎么行为"的**低维向量,与质量正交**。对 kernel,例如 `(算法族, 算术强度, 共享内存占用, 布局)`。
> 关键区分:fitness 回答"多好",BD 回答"哪一类"。**拿延迟当 BD,QD 就废了。**

### A.3 MAP-Elites 算法(最经典的 QD)
把行为空间**离散成网格**,每个格子(niche)只存一个**精英 = 该行为下找到的最快 kernel**:
```
archive = {}                                  # cell -> (kernel, latency)
for k in 经验池种子: place(archive, k)         # 用已知 kernel 初始化各格子(地板)

repeat:
    parent = 随机取一个已填格子的精英
    child  = LLM_vary(parent)                  # LLM 变异/重组 → 新 kernel
    lat    = measure(inner_param_search(child))# 实测(零成本)= 质量
    bd     = behavior_descriptor(child)        # 算术强度/内存/算法族/布局...
    cell   = bin(bd)
    if cell 为空 or lat < archive[cell].latency:
        archive[cell] = (child, lat)           # ★只和"同格子"竞争★
```
跑下来,archive 被一柜**多样且各自最优**的 kernel 填满。

### A.4 为什么多样性是"构造性保证"的(最要害)
看加 ★ 的那行:**竞争只发生在格子内部,不是全局**。
- 一个**新颖但稍慢**的 kernel,只要落在**空格子**就能存活(它是该 niche 的最好)。
- 在全局优化里它会因"不如当前最优"被直接丢掉。
- 具体到本场景:**Winograd kernel 与 direct kernel 落在不同"算法族"格子,彼此不竞争** → 即使某 shape 上 direct 更快,Winograd 这条线**不会被淘汰**,继续被探索改进。**这从机制上根除"过拟合到已知配方"**,也正是"经验池当地板不过滤"在算法层的体现。

### A.5 档案结构:grid vs CVT
- **网格(grid)**:BD 维度低(2–4)时直接用,简单直观。
- **CVT-MAP-Elites**:BD 维度高 / 边界未知时,用 **k 个质心(Centroidal Voronoi Tessellation)** 代替规则网格,避免维度爆炸。
- 预期会有"结构上达不到"的空格(如算法族与 AI 相关),正常,不影响。

### A.6 Novelty Search(QD 的另一支)
极端版:**(部分)抛掉 fitness,只奖励"行为上的新颖"**——离已见行为越远越好。纯 novelty 只求不同;QD = novelty + quality。实践中常把"新颖性"作为一种**选择压力**混入(对应 7.8 的 novelty 注入)。

### A.7 与 UCT / MCTS 的区别

| 维度 | UCT / MCTS | MAP-Elites / QD |
|---|---|---|
| 目标 | **单一最优**(最大化根价值) | **一整片多样精英档案** |
| 范式 | 收敛(convergent) | 发散(divergent) |
| 结构 | 增量生长的搜索树,节点=部分决策 | 行为空间档案,cell=行为生态位 |
| 竞争范围 | **全局**(价值回传、选最优路径) | **局部**(只与同 cell 精英比) |
| 多样性 | 靠探索项 `c√(lnN/n)` **临时**维持,最终仍收敛到一峰 | **构造性、永久**保留,本质多生态位 |
| 探索机制 | UCB 置信上界 | 空 cell 自然吸纳 + novelty 选择压 |
| 记忆 | 树(访问次数 + 价值) | 档案(每 cell 最优) |
| 评估需求 | 中(可用 rollout 估计) | **高**(填整个档案)→ 需廉价评估 |
| 输出 | 一条最优配置 | **一柜多样配置**(可多带 + 派发) |
| 防过拟合 | 探索项缓解,但本质单峰 | **本质多峰,根除塌缩** |
| 是否需 BD | 否 | **是(成败关键)** |

**一句话**:MCTS 是"用探索避免局部最优、最终仍求一个答案"的**收敛**搜索;QD 是"刻意维持铺开、永不塌缩"的**发散**搜索。本场景要覆盖、要抗过拟合、要为派发产出多份 kernel → QD 的发散性正中需求,而它"吃评估"的代价被你的零成本实测抵消。

### A.8 在本系统中的落地映射
| QD 概念 | 本系统对应 |
|---|---|
| 解(genotype) | 一个 kernel(结构 + 参数) |
| fitness | 内层参数搜索得到的**实测延迟** |
| 行为描述符 BD | 7.9 选定的轴(如 算法族 × AI × 布局) |
| 变异/重组算子 | **LLM**(受经验池 + roofline + novitly 引导) |
| 档案初始化 | **经验池种子**(地板,不过滤) |
| 剪枝/停机 | **roofline**(B&B 下界 + 逼近天花板停) |
| 产出 | 一张"行为 niche → 最优 kernel"地图,本身即不断增富的经验池,供跨设备派发 |

**代价与划算**:QD 的唯一硬伤是"要大量评估";本场景**实测零成本** → 硬伤消失。故 QD 对评估受限者不可行、**对本场景恰好可行**,这正是其差异化根基。
