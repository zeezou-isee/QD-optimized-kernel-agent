# **内层微观参数优化 (Micro-Level Hyperparameter Tuning) 方案设计**

在“宏微观双层解耦搜索”架构中，当 MAP-Elites（外层）固定了代码骨架（AST 结构）后，算子代码会被转化为一个带有参数占位符（如 <BLOCK_M>, <UNROLL_K>）的**参数化模板 (Parameterized Template)**。内层的任务就是对这个模板进行快速调优。

## **一、 核心参考文献（来自上传文件）**

您完全可以参考以下两篇您资料库中的论文，它们为内层参数调优提供了最直接的理论支撑：

1. **首选参考：《A Two-Stage GPU Kernel Tuner Combining Semantic Refactoring and Search-Based Optimization》 (Qu 等, 2026)**  
   * **核心思想**：这篇论文提出了与我们构想极其相似的“两阶段”框架。第一阶段（大模型）负责“语义重构（Semantic Refactoring）”生成模板；**第二阶段（传统算法）在硬件约束下对模板参数进行基于搜索的细粒度调优（Search-Based Optimization）。**  
   * **对您的启发**：它证明了“LLM 只负责写结构，传统算法负责填数字”是目前打破算子优化瓶颈的最有效途径。  
2. **次选参考：《KernelBand: Steering LLM-based Kernel Optimization via Hardware-Aware Multi-Armed Bandits》 (Ran 等, 2026)**  
   * **核心思想**：将内核优化视为多臂老虎机（MAB）问题，并设计了**基于硬件 Profiling 边界的剪枝策略（Hardware-aware pruning）**。  
   * **对您的启发**：在移动端，盲目搜索参数极易导致编译失败或 OOM。利用该论文中的边界剪枝思想，可以在内层搜索开始前砍掉大量无效参数。

## **二、 内层参数优化的 3 种主流方法**

针对移动端“真机测试成本极高”的特点，以下是三种按推荐程度排序的参数优化算法：

### **方法 1：基于硬件物理方程的 LLM 先验剪枝 + 树结构 Parzen 估计器 (TPE) 【🌟最推荐】**

这是目前深度学习编译器（如 TVM/Ansor）中超参数调优的工业界标杆，结合 LLM 会产生降维打击。

* **第一步：LLM 边界推导（Hardware Prior）**  
  * 不让 Agent 盲目猜参数，而是让 Agent 写出**物理约束方程**。  
  * *例子*：LLM 知道当前移动端 GPU 的共享内存 (Local Memory) 上限是 32KB。LLM 自动提取约束：TILE_M * TILE_N * 4 (bytes) <= 32768。  
  * 这直接将参数空间 [16, 32, 64, 128] 剪枝为合法的子集。  
* **第二步：TPE 算法搜索 (Tree-structured Parzen Estimator)**  
  * 在合法的参数子集中，使用 TPE 算法（一种高效的贝叶斯优化算法，Optuna 或 Hyperopt 库原生支持）进行采样。  
  * TPE 能够根据前几次真机测试的结果，迅速建立参数与 Latency 之间的概率模型，并将后续的采样集中在“大概率会很快”的参数区间。

### **方法 2：多保真度贝叶斯优化 (Multi-Fidelity Bayesian Optimization / BOHB)**

如果您有多个维度的参数（比如既有空间分块 Tiling，又有循环展开 Unroll，还有线程调度 Thread/Warp Size），真机测试依然很慢。

* **原理**：结合了 Hyperband（连续减半机制）和贝叶斯优化。  
* **移动端适配**：  
  * 生成 10 组不同的参数配置。  
  * **低保真度测试**：在宿主 PC 的 CPU 上用极为精简的框架模拟跑一遍（只验证数组不越界）。淘汰 5 组出错的。  
  * **中保真度测试**：推送到真机，但 Tensor Size 设为真实的 1/10。运行极短的时间。淘汰剩下的 3 组。  
  * **高保真度测试**：最后留下的 2 组最佳参数，使用完整 Tensor 跑完 100 次 iterations 并记录 Nsight / Simpleperf 级别的 Profiling。

### **方法 3：硬件感知的 UCB 多臂老虎机 (Hardware-Aware MAB)**

参考 *KernelBand* 论文的方法。

* **原理**：将每一个离散的参数选择（如 Unroll=2, Unroll=4）视为一台老虎机的摇臂。  
* **操作**：利用 UCB（置信度上限）公式，平衡对已知好参数的“剥削”和对未测试参数的“探索”。如果某组参数在过去的类似算子（通过 Memory 检索）中表现好，其先验 UCB 值会被 LLM 调高，从而优先被测试。

## **三、 MoKA 内层优化的伪代码工作流**
```
def optimize_inner_parameters(macro_template_code, hardware_specs):  
    """  
    MoKA 内层参数优化机制  
    :param macro_template_code: MAP-Elites 传递下来的带有 <PARAM> 占位符的 C++ 源码  
    :param hardware_specs: 移动端设备的物理限制 (Cache大小, 寄存器数量等)  
    """  
      
    # 1. 解析参数空间 (利用 LLM 将代码占位符映射为搜索空间)  
    search_space = LLM_Agent.extract_search_space(macro_template_code)  
    # 例如: {"TILE_M": [16, 32, 64, 128], "UNROLL_K": [1, 2, 4, 8]}  
      
    # 2. 物理先验剪枝 (参考 Qu et al. 和 KernelBand)  
    feasible_space = hardware_aware_prune(search_space, hardware_specs)  
    # 例如: 删除 TILE_M=128, 因为它超过了 ARM L1 Cache 上限  
      
    # 3. 初始化贝叶斯优化器 (如 TPE)  
    optimizer = TPE_Optimizer(space=feasible_space)  
      
    best_latency = float('inf')  
    best_params = None  
      
    # 极小预算的微观搜索 (例如 10 轮)  
    for i in range(10):  
        # TPE 推荐下一组最有潜力的参数  
        params = optimizer.ask()   
          
        # 将参数注入宏观模板  
        concrete_code = instantiate_template(macro_template_code, params)  
          
        # 编译与低保真度/高保真度评估  
        latency, compile_success = device_evaluator.evaluate(concrete_code)  
          
        if not compile_success:  
            # 严重惩罚，指导 TPE 避开该参数区  
            optimizer.tell(params, penalty=10000)   
        else:  
            # 记录成功结果  
            optimizer.tell(params, latency)  
            if latency < best_latency:  
                best_latency = latency  
                best_params = params  
                  
    # 4. 返回微观极限性能给外层 MAP-Elites  
    return best_params, best_latency  
```
